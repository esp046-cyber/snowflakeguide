üåü What is Snowflake?

Snowflake is a cloud-based data platform that allows you to store, process, and analyze data at scale. Think of it as a powerful database in the cloud that can handle massive amounts of data.

Key Features:

  • Cloud-Native: Runs entirely in the cloud (AWS, Azure, GCP)
  • Scalable: Automatically scales up or down based on your needs
  • Separation of Storage & Compute: Pay only for what you use
  • Zero Maintenance: No infrastructure to manage
  • Secure: Enterprise-grade security built-in
üí° Beginner Tip: Snowflake uses standard SQL, so if you know SQL, you're already halfway there!

Architecture Overview

Three Layers:

1. Storage Layer: Where your data lives
2. Compute Layer: Virtual warehouses that process queries
3. Cloud Services: Manages authentication, metadata, optimization

üõçÔ∏è Practice Query 1: Customer Data

Explore customer information from the TPCH dataset:

-- See top 10 customers SELECT C_CUSTKEY as customer_id, C_NAME as customer_name, C_MKTSEGMENT as market_segment, C_NATIONKEY as nation FROM SNOWFLAKE_SAMPLE_DATA.TPCH_SF1.CUSTOMER LIMIT 10;

üì¶ Practice Query 2: Order Analysis

Analyze orders by status:

-- Count orders by status SELECT O_ORDERSTATUS as order_status, COUNT(*) as total_orders, SUM(O_TOTALPRICE) as total_revenue FROM SNOWFLAKE_SAMPLE_DATA.TPCH_SF1.ORDERS GROUP BY O_ORDERSTATUS ORDER BY total_revenue DESC;

üí∞ Practice Query 3: Revenue by Year

Calculate annual revenue trends:

-- Revenue by year SELECT YEAR(O_ORDERDATE) as order_year, COUNT(*) as num_orders, ROUND(SUM(O_TOTALPRICE), 2) as total_revenue, ROUND(AVG(O_TOTALPRICE), 2) as avg_order_value FROM SNOWFLAKE_SAMPLE_DATA.TPCH_SF1.ORDERS GROUP BY order_year ORDER BY order_year DESC;

üåç Practice Query 4: Geographic Analysis

Join customers with nations:

-- Customers by country SELECT N.N_NAME as country, COUNT(C.C_CUSTKEY) as customer_count, R.R_NAME as region FROM SNOWFLAKE_SAMPLE_DATA.TPCH_SF1.CUSTOMER C JOIN SNOWFLAKE_SAMPLE_DATA.TPCH_SF1.NATION N ON C.C_NATIONKEY = N.N_NATIONKEY JOIN SNOWFLAKE_SAMPLE_DATA.TPCH_SF1.REGION R ON N.N_REGIONKEY = R.R_REGIONKEY GROUP BY country, region ORDER BY customer_count DESC LIMIT 10;

‚õÖ Practice Query 5: Weather Data

Query global weather information:

-- Weather observations SELECT country, postal_code, date_valid_std, avg_temperature_air_2m_f, avg_wind_speed_10m_mph FROM SNOWFLAKE_SAMPLE_DATA.WEATHER.HISTORY_DAY WHERE country = 'US' AND postal_code = '10001' ORDER BY date_valid_std DESC LIMIT 10;

üìä Practice Query 6: Advanced Aggregation

Find top selling products:

-- Top products by quantity sold SELECT L.L_PARTKEY as part_id, SUM(L.L_QUANTITY) as total_quantity, COUNT(DISTINCT L.L_ORDERKEY) as num_orders, ROUND(SUM(L.L_EXTENDEDPRICE), 2) as total_sales FROM SNOWFLAKE_SAMPLE_DATA.TPCH_SF1.LINEITEM L GROUP BY part_id ORDER BY total_sales DESC LIMIT 20;

üìä Data Types Guide

Understanding data types is crucial for creating tables and working with data efficiently.

üî§ String/Text Types

VARCHAR(n) or STRING

Stores text data of any length

-- Creating columns with VARCHAR CREATE TABLE users ( username VARCHAR(50), email VARCHAR(100), bio STRING -- No limit needed ); -- Examples INSERT INTO users VALUES ('john_doe', 'john@email.com', 'Software developer');

CHAR(n)

Fixed-length text (less common, use VARCHAR instead)

üî¢ Numeric Types

NUMBER(precision, scale)

Stores integers and decimals. Most versatile numeric type.

Precision = total digits, Scale = digits after decimal

CREATE TABLE products ( product_id NUMBER(10, 0), -- Integer, up to 10 digits price NUMBER(10, 2), -- Price: 99999999.99 weight NUMBER(8, 3), -- Weight: 12345.678 quantity NUMBER -- Default: NUMBER(38, 0) ); INSERT INTO products VALUES (1001, 29.99, 1.250, 100);

INTEGER / INT

Whole numbers only (alias for NUMBER(38,0))

CREATE TABLE inventory ( item_id INTEGER, stock_count INT );

FLOAT / DOUBLE

Approximate decimal numbers (scientific calculations)

üìÖ Date and Time Types

DATE

Stores year, month, day (YYYY-MM-DD)

CREATE TABLE events ( event_date DATE ); INSERT INTO events VALUES ('2026-01-21'), (CURRENT_DATE()), (TO_DATE('01/21/2026', 'MM/DD/YYYY'));

TIME

Stores hours, minutes, seconds (HH:MI:SS)

CREATE TABLE schedules ( start_time TIME ); INSERT INTO schedules VALUES ('14:30:00'), (CURRENT_TIME());

TIMESTAMP

Stores date + time (most commonly used for timestamps)

CREATE TABLE logs ( created_at TIMESTAMP, updated_at TIMESTAMP_NTZ, -- No timezone logged_at TIMESTAMP_LTZ, -- Local timezone server_time TIMESTAMP_TZ -- With timezone ); INSERT INTO logs VALUES (CURRENT_TIMESTAMP(), '2026-01-21 14:30:00', CURRENT_TIMESTAMP(), CURRENT_TIMESTAMP());
üí° Best Practice: Use TIMESTAMP_NTZ for most cases - it's timezone-agnostic and simplest to work with.

‚úÖ Boolean Type

BOOLEAN

TRUE or FALSE values

CREATE TABLE settings ( feature_name VARCHAR(50), is_enabled BOOLEAN ); INSERT INTO settings VALUES ('dark_mode', TRUE), ('notifications', FALSE), ('auto_save', NULL); -- Can be NULL

üé® Semi-Structured Types

VARIANT

Stores JSON, XML, or any semi-structured data. Very powerful!

CREATE TABLE api_responses ( response_id INT, data VARIANT ); -- Insert JSON data INSERT INTO api_responses VALUES (1, PARSE_JSON('{"name":"John","age":30,"active":true}')); -- Query JSON fields SELECT data:name::STRING as name, data:age::INT as age, data:active::BOOLEAN as active FROM api_responses;

OBJECT

Stores JSON objects specifically

ARRAY

Stores JSON arrays

CREATE TABLE collections ( tags ARRAY, scores ARRAY ); INSERT INTO collections VALUES (ARRAY_CONSTRUCT('tech', 'news', 'blog'), ARRAY_CONSTRUCT(95, 87, 92));

üíæ Binary Type

BINARY

Stores binary data (files, images as bytes)

üîÑ Type Conversion

-- Convert between types SELECT CAST('123' AS INTEGER), '123'::INTEGER, -- Shorthand TO_NUMBER('123.45'), TO_DATE('2026-01-21'), TO_TIMESTAMP('2026-01-21 14:30:00'), TO_VARCHAR(12345), TO_BOOLEAN('true'); -- Safe conversion (returns NULL if fails) SELECT TRY_CAST('abc' AS INTEGER), -- Returns NULL TRY_TO_NUMBER('not_a_number'); -- Returns NULL

üìè Type Comparison Table

For Names/Emails: VARCHAR or STRING
For Prices: NUMBER(10,2)
For Counts: INTEGER
For Dates: DATE
For Timestamps: TIMESTAMP_NTZ
For JSON Data: VARIANT
For Yes/No: BOOLEAN

-- E-commerce table example CREATE TABLE orders ( order_id INTEGER, customer_email VARCHAR(100), order_date DATE, order_timestamp TIMESTAMP_NTZ, total_amount NUMBER(10, 2), is_paid BOOLEAN, items VARIANT, -- JSON array of items metadata VARIANT -- Additional JSON data );

üë• Roles & Permissions

Roles control what you can do in Snowflake. Think of them as different levels of access.

üí° Key Concept: Always use the least powerful role needed for your task. Don't stay in ACCOUNTADMIN!

üîë Main System Roles

ACCOUNTADMIN

Power Level: ‚≠ê‚≠ê‚≠ê‚≠ê‚≠ê (Highest)

What it does: Complete control over the entire account

Can do:

  • Manage users and roles
  • View and control billing
  • Create/drop databases and warehouses
  • Everything SYSADMIN can do
  • Change account settings

When to use: Account setup, user management, billing review

‚ö†Ô∏è Warning: Don't use for daily work! Very risky.

SYSADMIN

Power Level: ‚≠ê‚≠ê‚≠ê‚≠ê (High)

What it does: Manage databases and warehouses

Can do:

  • Create databases, schemas, tables
  • Create and manage warehouses
  • Grant privileges to other roles
  • Cannot manage users or billing

When to use: Creating databases, setting up data structures

‚úì Recommended: Use this for most admin work

-- Switch to SYSADMIN USE ROLE SYSADMIN; -- Create database and warehouse CREATE DATABASE analytics_db; CREATE WAREHOUSE analytics_wh WITH WAREHOUSE_SIZE = 'SMALL';

SECURITYADMIN

Power Level: ‚≠ê‚≠ê‚≠ê

What it does: Manage users and roles (but not account settings)

Can do:

  • Create and manage roles
  • Grant roles to users
  • Manage privileges

When to use: Setting up permissions, creating custom roles

PUBLIC

Power Level: ‚≠ê (Lowest)

What it does: Basic access granted to all users

Can do:

  • Access objects granted to PUBLIC
  • Very limited permissions

When to use: Default minimal access for all users

üéØ Role Hierarchy

ACCOUNTADMIN (Top - owns everything) ‚îî‚îÄ‚îÄ SECURITYADMIN (Manages users/roles) ‚îî‚îÄ‚îÄ SYSADMIN (Manages databases/warehouses) ‚îî‚îÄ‚îÄ Custom Roles (Your created roles) ‚îî‚îÄ‚îÄ PUBLIC (Bottom - everyone has this)

üìù Creating Custom Roles

For teams, create specific roles with only needed permissions:

-- Switch to SECURITYADMIN to create roles USE ROLE SECURITYADMIN; -- Create a role for analysts CREATE ROLE analyst_role; -- Grant warehouse usage GRANT USAGE ON WAREHOUSE compute_wh TO ROLE analyst_role; -- Grant database access GRANT USAGE ON DATABASE analytics_db TO ROLE analyst_role; -- Grant schema access GRANT USAGE ON SCHEMA analytics_db.public TO ROLE analyst_role; -- Grant SELECT on all tables GRANT SELECT ON ALL TABLES IN SCHEMA analytics_db.public TO ROLE analyst_role; -- Assign role to a user GRANT ROLE analyst_role TO USER jane_doe;

üîç Checking Your Current Role

-- See your current role SELECT CURRENT_ROLE(); -- See all your available roles SHOW ROLES; -- See what roles you have SELECT * FROM TABLE(RESULT_SCAN(LAST_QUERY_ID()));

üîÑ Switching Roles

-- Switch to different role USE ROLE SYSADMIN; -- Or in same statement USE ROLE ACCOUNTADMIN; CREATE DATABASE my_db; USE ROLE SYSADMIN; -- Switch back immediately

‚ö° Common Role Use Cases

Setting up account initially: ACCOUNTADMIN

Creating databases/warehouses: SYSADMIN

Creating users and roles: SECURITYADMIN

Daily analytics work: Custom analyst role

Running queries: Role with SELECT privileges

üõ°Ô∏è Security Best Practices

  • ‚úì Use SYSADMIN for daily admin work, not ACCOUNTADMIN
  • ‚úì Create custom roles for different teams
  • ‚úì Grant minimum necessary privileges
  • ‚úì Regularly review who has which roles
  • ‚úó Don't give everyone ACCOUNTADMIN
  • ‚úó Don't stay logged in as ACCOUNTADMIN
‚ö†Ô∏è Security Tip: Only 2-3 trusted people should have ACCOUNTADMIN. Everyone else should use SYSADMIN or custom roles.

üìã Example: Setting Up a Data Team

-- As SECURITYADMIN, create team roles USE ROLE SECURITYADMIN; -- Analysts: Read-only access CREATE ROLE data_analyst; GRANT USAGE ON WAREHOUSE analytics_wh TO ROLE data_analyst; GRANT USAGE ON DATABASE prod_db TO ROLE data_analyst; GRANT SELECT ON ALL TABLES IN SCHEMA prod_db.public TO ROLE data_analyst; -- Engineers: Can modify data CREATE ROLE data_engineer; GRANT USAGE ON WAREHOUSE etl_wh TO ROLE data_engineer; GRANT ALL ON DATABASE staging_db TO ROLE data_engineer; -- Assign to users GRANT ROLE data_analyst TO USER analyst1; GRANT ROLE data_engineer TO USER engineer1;
üí° Remember: You can have multiple roles, but only one is active at a time. Switch roles based on what task you're doing!

üè≠ Virtual Warehouses

Virtual warehouses are Snowflake's compute engines. They process your queries and load data.

Key Concepts:

  • Size: X-Small to 6X-Large (affects speed and cost)
  • Auto-Suspend: Pauses when not in use (saves money!)
  • Auto-Resume: Starts automatically when needed
  • Multi-Cluster: Scales out for heavy workloads

Creating a Warehouse

CREATE WAREHOUSE my_warehouse WITH WAREHOUSE_SIZE = 'X-SMALL' AUTO_SUSPEND = 60 AUTO_RESUME = TRUE;
‚ö†Ô∏è Cost Tip: Always set AUTO_SUSPEND to avoid unnecessary charges. Warehouses bill per second!

Warehouse Sizes & Use Cases

X-Small: Testing, small queries
Small: Development work
Medium: Regular analytics
Large+: Heavy processing, complex queries

Managing Warehouses

-- Start a warehouse ALTER WAREHOUSE my_warehouse RESUME; -- Pause a warehouse ALTER WAREHOUSE my_warehouse SUSPEND; -- Resize a warehouse ALTER WAREHOUSE my_warehouse SET WAREHOUSE_SIZE = 'MEDIUM';

‚è∞ Time Travel & Fail-Safe

One of Snowflake's most powerful features - the ability to access and restore historical data!

üí° Amazing Feature: Accidentally deleted data? Updated the wrong rows? Time Travel can save you!

üïê What is Time Travel?

Time Travel lets you query, clone, and restore data from the past. Think of it as an "undo" button for your database.

Standard Edition: 1 day of Time Travel
Enterprise Edition: Up to 90 days
Default: 1 day for all accounts

üìñ Query Historical Data

See what your data looked like at a specific time:

-- Query data as it was 1 hour ago SELECT * FROM orders AT(OFFSET => -60*60); -- 60 seconds * 60 minutes -- Query data as of specific timestamp SELECT * FROM orders AT(TIMESTAMP => '2026-01-21 10:00:00'::TIMESTAMP); -- Query before a specific statement SELECT * FROM orders BEFORE(STATEMENT => '01a9e15f-0000-1234-5678-90abcdef1234');

üîÑ Restore Deleted Tables

Recover a table you accidentally dropped:

-- Oops! Dropped the wrong table DROP TABLE customers; -- No problem! Restore it UNDROP TABLE customers; -- Works for databases and schemas too UNDROP DATABASE my_database; UNDROP SCHEMA my_schema;
‚ö†Ô∏è Time Limit: UNDROP only works within the Time Travel retention period (1-90 days depending on edition).

‚Ü©Ô∏è Restore Data After Wrong Update

Fix data after a bad UPDATE or DELETE:

-- Oh no! Accidentally updated all prices to 0 UPDATE products SET price = 0; -- Method 1: Create new table from historical data CREATE TABLE products_restored AS SELECT * FROM products AT(OFFSET => -60*5); -- 5 minutes ago -- Method 2: Insert historical data back INSERT INTO products SELECT * FROM products BEFORE(STATEMENT => '01a9e15f-0000-1234'); -- Method 3: Replace entire table CREATE OR REPLACE TABLE products AS SELECT * FROM products AT(TIMESTAMP => '2026-01-21 14:00:00'::TIMESTAMP);

üìã Clone Tables from History

Create instant copies of historical data (zero-copy clone):

-- Clone table as it was yesterday CREATE TABLE orders_backup CLONE orders AT(OFFSET => -60*60*24); -- Clone before a specific change CREATE TABLE customers_safe CLONE customers BEFORE(STATEMENT => '01a9e15f-0000-1234'); -- Clone entire database CREATE DATABASE prod_backup CLONE production_db AT(TIMESTAMP => '2026-01-20 23:59:59'::TIMESTAMP);
üí° Zero-Copy Magic: Clones don't duplicate storage initially - they only store differences. Super fast and efficient!

üîç Finding Query IDs for Time Travel

-- View your recent query history SELECT query_id, query_text, start_time, end_time FROM TABLE(INFORMATION_SCHEMA.QUERY_HISTORY()) WHERE query_type = 'UPDATE' OR query_type = 'DELETE' ORDER BY start_time DESC LIMIT 10; -- Use the query_id in BEFORE(STATEMENT => ...)

‚öôÔ∏è Configure Time Travel Retention

-- Set retention when creating table (Enterprise only) CREATE TABLE important_data ( id INT, value VARCHAR ) DATA_RETENTION_TIME_IN_DAYS = 90; -- Change retention on existing table ALTER TABLE customers SET DATA_RETENTION_TIME_IN_DAYS = 7; -- Check current retention SHOW TABLES LIKE 'customers';

üõ°Ô∏è Fail-Safe: The Last Resort

After Time Travel period ends, data enters Fail-Safe (additional 7 days).

Fail-Safe Period: 7 days (automatic)
Purpose: Disaster recovery only
Access: Must contact Snowflake Support
Cost: Requires support engagement

üìä Complete Timeline

Day 0: Data created/modified ‚Üì Days 1-90: TIME TRAVEL (you can query/restore) ‚Üì Days 91-97: FAIL-SAFE (Snowflake Support only) ‚Üì Day 98+: Data permanently deleted

üí∞ Storage Costs

Time Travel and Fail-Safe data count toward storage costs:

-- View Time Travel storage usage SELECT * FROM SNOWFLAKE.ACCOUNT_USAGE.TABLE_STORAGE_METRICS WHERE TABLE_NAME = 'ORDERS' ORDER BY ACTIVE_BYTES DESC;
‚ö†Ô∏è Cost Tip: Longer Time Travel = more storage costs. Use 90 days only for critical tables.

üéØ Common Use Cases

Accidental DELETE: Restore with AT/BEFORE

Bad UPDATE: Clone historical version

Dropped TABLE: UNDROP immediately

Testing Changes: Query before/after comparison

Auditing: See what data looked like historically

Point-in-time Backup: Clone database nightly

üìù Best Practices

  • ‚úì Test UNDROP and Time Travel queries in development first
  • ‚úì Document important query_ids after major changes
  • ‚úì Use shorter retention (1-7 days) for non-critical tables
  • ‚úì Create clones before major updates as safety net
  • ‚úì Monitor Time Travel storage usage regularly
  • ‚úó Don't rely on Time Travel as your only backup strategy

üöÄ Real-World Example

-- Friday 5pm: Major data update goes wrong UPDATE customer_orders SET status = 'cancelled' WHERE order_date > '2026-01-01'; -- Oops! Should have been order_date < '2026-01-01' -- Friday 5:05pm: Realize the mistake -- Get the query ID SELECT query_id, query_text, start_time FROM TABLE(INFORMATION_SCHEMA.QUERY_HISTORY()) WHERE query_text LIKE '%UPDATE customer_orders%' ORDER BY start_time DESC LIMIT 1; -- Returns: query_id = '01a9e15f-0000-1234' -- Restore data to state before bad update CREATE OR REPLACE TABLE customer_orders AS SELECT * FROM customer_orders BEFORE(STATEMENT => '01a9e15f-0000-1234'); -- Crisis averted! ‚úÖ
üí° Pro Tip: Before running risky UPDATE/DELETE statements, note the current timestamp. Makes recovery much easier!

üîß Common Errors & Solutions

Don't worry - everyone gets errors! Here's how to fix the most common ones.

üí° Quick Tip: Read error messages carefully - they usually tell you exactly what's wrong!

‚ùå Error 1: No Active Warehouse

SQL compilation error: No active warehouse selected

What it means: You haven't selected a warehouse to run your query.

Solution:

-- Select a warehouse first USE WAREHOUSE COMPUTE_WH; -- Or create one if none exists CREATE WAREHOUSE my_warehouse WITH WAREHOUSE_SIZE = 'X-SMALL' AUTO_SUSPEND = 60 AUTO_RESUME = TRUE; USE WAREHOUSE my_warehouse;

‚ùå Error 2: Warehouse Not Running

Warehouse 'COMPUTE_WH' is suspended. Resuming...

What it means: The warehouse is paused (this is good for saving money!).

Solution: Just wait a few seconds - Snowflake will auto-resume it if AUTO_RESUME is enabled. Or manually resume:

ALTER WAREHOUSE COMPUTE_WH RESUME;

‚ùå Error 3: Permission Denied

SQL access control error: Insufficient privileges to operate on database 'MY_DB'

What it means: Your current role doesn't have permission.

Solution:

-- Switch to a role with more permissions USE ROLE ACCOUNTADMIN; -- Or use SYSADMIN for creating objects USE ROLE SYSADMIN; -- Check your current role SELECT CURRENT_ROLE();

‚ùå Error 4: Object Does Not Exist

SQL compilation error: Object 'CUSTOMERS' does not exist

What it means: The table, database, or schema you're trying to use doesn't exist.

Solution:

-- Check if you're in the right database/schema SELECT CURRENT_DATABASE(), CURRENT_SCHEMA(); -- Set the correct database USE DATABASE my_database; USE SCHEMA my_schema; -- List available tables SHOW TABLES; -- Or use full path SELECT * FROM my_database.my_schema.customers;

‚ùå Error 5: Syntax Error

SQL compilation error: syntax error line 1 at position 15 unexpected 'FORM'

What it means: You have a typo in your SQL command.

Common mistakes:

  • FROM spelled as FORM
  • Missing commas between column names
  • Forgetting semicolons at the end
  • Unclosed quotes or parentheses
-- Wrong ‚ùå SELECT name, email FORM customers; -- Correct ‚úì SELECT name, email FROM customers;

‚ùå Error 6: Column Not Found

SQL compilation error: invalid identifier 'EMIAL'

What it means: Column name is misspelled or doesn't exist.

Solution:

-- Check actual column names DESCRIBE TABLE customers; -- Or view sample data SELECT * FROM customers LIMIT 5;

‚ùå Error 7: Date/Time Format Issues

Date '2026-13-01' is not recognized

What it means: Invalid date format (month 13 doesn't exist).

Solution:

-- Correct date format: YYYY-MM-DD SELECT * FROM orders WHERE order_date = '2026-01-13'; -- Convert string to date SELECT TO_DATE('01/13/2026', 'MM/DD/YYYY'); -- Common date functions SELECT CURRENT_DATE(); SELECT DATEADD(day, 7, CURRENT_DATE());

‚ùå Error 8: Data Type Mismatch

Numeric value 'abc' is not recognized

What it means: Trying to use text where a number is expected.

Solution:

-- Wrong ‚ùå SELECT * FROM orders WHERE price = 'expensive'; -- Correct ‚úì SELECT * FROM orders WHERE price > 100; -- Convert types if needed SELECT CAST('123' AS INTEGER); SELECT TRY_CAST('abc' AS INTEGER); -- Returns NULL if fails

‚ùå Error 9: File Not Found (Loading Data)

File 'data.csv' does not exist or not authorized

What it means: File path is wrong or file wasn't uploaded to stage.

Solution:

-- List files in stage LIST @my_stage; -- Check stage exists SHOW STAGES; -- Make sure file is uploaded to the stage first

‚ùå Error 10: Quota Exceeded

Query exceeded warehouse size limit

What it means: Query is too big for your warehouse size.

Solution:

-- Increase warehouse size temporarily ALTER WAREHOUSE my_warehouse SET WAREHOUSE_SIZE = 'MEDIUM'; -- Or optimize your query with LIMIT SELECT * FROM huge_table LIMIT 1000; -- Add WHERE clause to filter data SELECT * FROM huge_table WHERE date > '2026-01-01';

‚ùå Error 11: Connection Timeout

Query timeout exceeded. Query was killed.

What it means: Query took too long to run.

Solution:

-- Use larger warehouse for complex queries ALTER WAREHOUSE my_warehouse SET WAREHOUSE_SIZE = 'LARGE'; -- Optimize query with indexes or filters -- Break into smaller chunks

‚ùå Error 12: Duplicate Key

Duplicate key value violates unique constraint

What it means: Trying to insert a value that already exists in a unique column.

Solution:

-- Check for existing records first SELECT * FROM customers WHERE id = 123; -- Use UPDATE instead of INSERT UPDATE customers SET email = 'new@email.com' WHERE id = 123; -- Or use MERGE for upsert behavior
‚ö†Ô∏è Still Stuck? Copy the full error message and search for it in Snowflake documentation or community forums!

üéØ Best Practices for Beginners

1. Cost Management

  • Always set AUTO_SUSPEND on warehouses (60 seconds is good)
  • Use the smallest warehouse size that meets your needs
  • Monitor your usage in the "Admin" ‚Üí "Usage" section
  • Suspend warehouses when done for the day
‚ö†Ô∏è Running a Medium warehouse 24/7 can cost ~$3,000/month. Always auto-suspend!

2. Query Optimization

  • Use SELECT specific columns instead of SELECT *
  • Filter data with WHERE clauses early
  • Use LIMIT when testing queries
  • Check query history for slow queries

3. Organization

-- Good structure: DATABASE ‚Üí SCHEMA ‚Üí TABLE Example: SALES_DB ‚Üí CUSTOMER_SCHEMA ‚Üí ORDERS_TABLE

4. Security Basics

  • Use strong passwords and enable MFA
  • Grant minimum necessary privileges
  • Don't use ACCOUNTADMIN for daily work
  • Create separate roles for different users
